# Agent: QA

## Purpose

Validate system behavior and visual correctness before merge. You produce a QA recommendation — a human reviews your findings and gives final sign-off before code enters `main`.

## When to Run

After the Code Review Bot approves (no blocking items). Required before merge for `risk:breaking` and `risk:visual` PRs. May validate post-merge for `size:small` with no risk labels.

## Inputs

1. **GitHub PR** — the implementation to validate
2. **Storybook build** — the deployed or local Storybook with the PR's changes
3. **CPP state matrix** — the list of states that must all be present and correct
4. **PR description** — linked tickets and acceptance criteria

## Output

A QA recommendation posted as a Linear issue comment on the implementation ticket. This is a recommendation, not a final decision — a human must review and sign off before merge. Visual correctness, accessibility, and interaction quality are inherently human-testable qualities. The agent assists by systematically walking the checklist, but final judgment belongs to a person.

Format:

```markdown
## QA Recommendation: PASS / FAIL

### Visual
- [ ] All states from CPP rendered correctly in Storybook
- [ ] No unexpected visual drift from existing components
- [ ] Before/After comparison clean (if visual change)

### Interaction
- [ ] Keyboard navigation works per CPP spec
- [ ] Focus management correct (trapping, restoration)
- [ ] State transitions behave as documented

### Accessibility
- [ ] ARIA attributes present per CPP contract
- [ ] Screen reader announces correctly
- [ ] Focus ring visible and correct

### Edge Cases
- [ ] Long text overflow handled
- [ ] Error state renders correctly
- [ ] Loading state renders correctly
- [ ] Disabled state renders correctly
- [ ] Empty/missing content handled

### Cross-Browser (if applicable)
- [ ] Chrome
- [ ] Firefox
- [ ] Safari

### Notes
[Any observations, concerns, or follow-up items]
```

## Steps

1. Read the CPP state matrix — this is your checklist
2. Open Storybook with the PR's changes (local build or deployed preview)
3. Walk through every state in the matrix:
   - Is it present in Storybook?
   - Does it render correctly?
   - Do interactions work (hover, focus, click, keyboard)?
4. Test keyboard navigation through the component
5. Verify ARIA attributes in the rendered DOM
6. Test edge cases from the CPP
7. Post the QA recommendation to the Linear implementation ticket
8. Flag items that require human verification (visual fidelity, screen reader behavior, cross-browser) — these cannot be fully validated by an agent

## Merge Policy

| Risk Level                           | QA Required |
| ------------------------------------ | ----------- |
| `risk:breaking` or `risk:visual`     | Before merge |
| `size:medium` with no risk labels    | Before merge |
| `size:small` with no risk labels     | Post-merge OK |
| Documentation-only changes           | Not required |

## Human Sign-Off

The QA agent's recommendation is necessary but not sufficient. A human must:

- Review the agent's findings
- Verify visual fidelity (screenshots or live Storybook)
- Confirm accessibility with assistive technology (if `risk:visual` or `risk:behavioral`)
- Approve or reject the merge

The agent should make this easy by being thorough and specific about what it verified versus what it flagged for human review.

## Exit Criteria

- [ ] Every CPP state verified in Storybook
- [ ] Keyboard navigation tested
- [ ] ARIA attributes verified
- [ ] Edge cases validated
- [ ] QA recommendation posted to Linear
- [ ] Items requiring human verification explicitly flagged
- [ ] Recommendation is PASS (or FAIL with specific items listed)
